# Azure Data Lake Storage Gen2 (ADLS Gen2)

## Phase 2 - Sprint 1 - Story 2

---

# Objective

Create an enterprise-grade Azure Data Lake Storage Gen2 (ADLS Gen2) environment that serves as the central storage layer for the Retail Analytics Platform following the Medallion Architecture (Bronze, Silver, Gold).

---

# What is Azure Data Lake Storage Gen2?

Azure Data Lake Storage Gen2 (ADLS Gen2) is Microsoft's enterprise cloud data lake built on Azure Blob Storage.

It combines the scalability and cost efficiency of Blob Storage with hierarchical file system capabilities, making it suitable for analytics workloads using Azure Data Factory, Azure Databricks, Synapse Analytics, and Power BI.

---

# Why ADLS Gen2?

The Retail Analytics Platform requires a storage solution that supports:

* Large-scale data ingestion
* Batch ETL pipelines
* Incremental data loading
* Hierarchical directory structures
* Secure cloud storage
* Integration with Azure Data Factory
* Integration with Azure Databricks

ADLS Gen2 satisfies all these requirements.

# Why Hierarchical Namespace?

Hierarchical Namespace enables ADLS Gen2 features such as:

* Folder hierarchy
* Atomic directory operations
* Improved analytics performance
* POSIX-style directory structure
* Better compatibility with Spark workloads

Without Hierarchical Namespace, the storage account behaves as a standard Azure Blob Storage account.

# Medallion Architecture

The Retail Analytics Platform follows the Medallion Architecture.

```
Raw Data
    │
    ▼
Bronze Layer
    │
    ▼
Silver Layer
    │
    ▼
Gold Layer
```

Each layer progressively improves data quality and business value.

---

# Data Lake Containers

The following containers were created.

| Container | Purpose                                |
| --------- | -------------------------------------- |
| bronze    | Raw source data                        |
| silver    | Cleansed and validated data            |
| gold      | Curated business-ready datasets        |
| metadata  | Pipeline metadata and configuration    |
| logs      | Pipeline execution and validation logs |
| archive   | Historical processed files             |
| rejected  | Invalid or rejected records            |
| $logs     | Azure system-generated logs            |

---

# Bronze Layer Structure

The Bronze layer stores raw data exactly as received from the source systems.

```
bronze
│
├── master
│   ├── customers
│   │   ├── full_load
│   │   └── incremental
│   │
│   ├── products
│   └── stores
│
└── transactions
    ├── orders
    │   ├── full_load
    │   └── incremental
    │
    └── order_items
        ├── full_load
        └── incremental
```

The Bronze layer preserves the original files without modification.

---

# Silver Layer Structure

The Silver layer contains validated, standardized, and cleansed datasets generated by Azure Databricks.

```
silver
│
├── master
│   ├── customers
│   ├── products
│   └── stores
│
└── transactions
    ├── orders
    └── order_items
```

Data quality checks, schema validation, and transformations are performed before loading data into this layer.

---

# Gold Layer Structure

The Gold layer contains business-ready datasets optimized for reporting and analytics.

```
gold
│
├── sales_mart
├── customer_mart
├── product_mart
└── executive_dashboard
```

These datasets will be consumed directly by Power BI dashboards.

# Supporting Containers

## Metadata

```
metadata
├── config
├── schemas
├── watermarks
└── pipeline_config
```

Stores configuration files, schema definitions, watermark information, and metadata used by Azure Data Factory.

---

## Logs

```
logs
├── adf
├── databricks
├── validation
└── audit
```

Stores execution logs, validation reports, audit logs, and monitoring information.

---

## Archive

```
archive
├── master
└── transactions
```

Stores processed files for historical reference and recovery.

---

## Rejected

```
rejected
├── master
└── transactions
```

Stores invalid files or records that fail validation rules.
